# フォルダ分け計画

## 計画
NotebookLM の内部データには干渉せず、FolderLM の割り当て情報と DOM 復帰（Recovery）に連携しながら、ノート一覧へフォルダ順の並べ替え/グループ化表示を追加する。NotebookLM 標準フィルタとの AND 条件を維持し、DOM/CSS のみで軽量かつ非侵襲に動作させる。

## 要件
- NotebookLM 標準フィルタとの AND 条件を維持する。
- DOM 表示のみを操作し、NotebookLM 内部データへは干渉しない。
- クリックターゲットやリストセマンティクスを壊さず、アクセシビリティに配慮する。
- ノート ID は `aria-labelledby="project-<UUID>-title"` から抽出する。
- フォルダ割り当ては **1ノート=1フォルダ**（`noteAssignments: { [noteId]: folderId }`）を前提とする。

## 決定事項
- グループヘッダー表示は「すべて」選択時のみ行う。
- `viewMode` は `storageManager.settings` に保存し、リロード後も維持する。
- MVP は **sort モード**を優先する。

## 表示モード定義
- `filter`: 現行と同様に非該当ノートを非表示（並べ替えなし）。
- `sort`: フォルダ順に並べ替え、ヘッダーなし。
  - 可能なら CSS `order` を使用し、不可なら DOM 並べ替えにフォールバック。
- `group`: フォルダ順に並べ替え + グループヘッダー表示。
  - 「すべて」選択時のみ有効、DOM 並べ替え + ヘッダー挿入。

## スコープ
- **範囲内**: 並べ替え/グループ化ロジック、グループヘッダー、順序変更、復帰処理への統合、表示モード用 UI。
- **範囲外**: NotebookLM 内部順序の永続変更、ドラッグ&ドロップ、バックグラウンド処理、データモデルの拡張。

## データモデル / 挙動
- グループ順序は `storageManager.getFolders()` の順（「未分類」を含む）を使用。
- 未割り当てノートは「未分類」に分類。
- 同一フォルダ内の順序は NotebookLM の元の DOM 順序を尊重する。
- 安定ソートのため、スキャン時に元のインデックスを記録（`dataset` または `WeakMap`）。
  - 再レンダリング/再スキャン時には必ず再初期化し、古い順序を引きずらない。

## 実装ポイント（ファイル/関数）

  - `selectFolder()` に `viewMode` 適用を統合
- `src/content/core/domRecoveryManager.js`
  - `_checkRecoveryNeeded()` でグループヘッダー/並べ替え状態も検知

**実装内容**:

- `_groupNotesByFolder()`: ソート済みノートをフォルダIDでグループ化し、フォルダ順序でグループ配列を返す
- `_applyGroupedDomReorder()`: グループヘッダーを挿入しながらノートをDOM並べ替え。「すべて」選択時のみヘッダーを表示
- `_createGroupHeader()`: グループヘッダー要素を生成。`role="presentation"`, `aria-hidden="true"`, `tabindex="-1"`, `inert` でアクセシビリティ対応

**アクセシビリティ対応**:
- `role="presentation"`: セマンティック上の意味を持たないことを示す
- `aria-hidden="true"`: スクリーンリーダーから非表示
- `tabindex="-1"`: キーボードフォーカス不可
- `inert`: 完全に非操作
- `pointer-events: none`: マウス操作不可

**実装内容**:
- `domRecoveryManager._startObservingSortChange()`: ノートリストコンテナを監視し、DOM 変化（ソート変更、仮想化）を検知
- `domRecoveryManager._handleSortMutations()`: グループヘッダー削除や複数カードの移動を検知して viewMode 再適用をトリガー
- `domRecoveryManager.setViewModeReapplyCallback()`: filterManager から viewMode 再適用処理を登録
- `domRecoveryManager.setSortChangeCallback()`: NotebookLM ソート変更時のコールバックを登録
- `filterManager.checkViewModeRecoveryNeeded()`: sort/group モードの状態をチェックし復帰が必要か判定
- `filterManager._checkSortRecoveryNeeded()`: ソート済みカードや order 属性の欠落を検知
- `filterManager._checkGroupRecoveryNeeded()`: グループヘッダーやグループ化済みカードの欠落を検知
- `filterManager._groupByFolderWithFallback()`: グループモード適用時にエラーが発生した場合、sort モードにフォールバック

**動作フロー**:
1. NotebookLM がノートリストを再描画（ソート変更、仮想化など）
2. `domRecoveryManager._sortObserver` が DOM 変化を検知
3. グループヘッダー削除や複数カード移動を検出した場合、`_debouncedReapplyViewMode` をトリガー
4. `_viewModeReapplyCallback` が呼び出され、元インデックスリセット → ノート再スキャン → フィルタ再適用
5. group モードで失敗が続く場合は自動的に sort モードにフォールバック

## パフォーマンス方針
- 更新は `requestAnimationFrame` でバッチ化し、連続変化を統合する。
- 差分検知で最小限の再並べ替え（全再構築を避ける）。
- 100 ノート程度を想定し、16ms を目標にする。

## リスクとエッジケース
- NotebookLM 側の DOM 構造変更（aria 属性/クラス変化）でセレクタが破綻する。
- 仮想化によりヘッダー/並び替えが消える可能性がある。
- CSS `order` が効かないレイアウトの場合、DOM 並べ替えで競合が起きる。
- グループヘッダー挿入でリストセマンティクス/A11y が崩れる可能性がある。
- フォルダ削除時、割り当てが「未分類」へ移動する前提で UI も同期する必要がある。
